## User Story

As a content publisher, I want to create nested MDX content files with frontmatter in a top-level `content` directory, so that root-level pages are published from content files without requiring code changes for each page.

## Goal

- Deliver an MDX-powered content publishing flow where publishers can add nested content files and have end-user pages available at root-level routes with approved reusable components.

## Approach

- Implement the feature in vertical slices: routing + content loading first, then MDX component support, then validation/publishing safeguards.
- Keep the authoring model file-based and deterministic from content path + frontmatter.
- Enforce quality gates so invalid content blocks CI while preserving local iteration via warnings.

## Phases

### Phase 1

- Changes: Add content-source conventions (`index.mdx` directory mapping), root-level route mapping for top-level + second-level pages via catch-all route in `app`, and frontmatter parsing/rendering for required fields.
- Validation: Confirm sample nested pages resolve and render frontmatter/body correctly; run lint/build checks.
- Tracker updates: Keep story status `Analysing` during plan and early implementation.

### Phase 2

- Changes: Add approved MDX components (video, links, Sendy signup), and implement invalid-content handling policy (CI fail, local/dev warn).
- Validation: Confirm components render in authored MDX, and invalid fixtures fail CI while warning locally.
- Tracker updates: Move through `In Progress` -> `Testing` -> `Complete` only after user-confirmed validation.

## Files / Areas Likely to Change

- `app/[...slug]/page.tsx` (or equivalent catch-all content route under `app/` for recursive root-level resolution)
- Content-loading/parsing utilities (new or existing helper area under `app/` or `lib/`)
- MDX component mapping (likely `components/` and shared content renderer)
- Top-level content source folder (new) for MDX files and nested directories
- Build/validation hooks for CI vs local/dev content validation behavior

## Data / RLS / RPC / Migrations

- No database schema, RLS, RPC, or migration changes are expected for this story.
- Content is file-based in-repo, so data change impact is limited to content source files and route/render pipeline behavior.

## Risks & Mitigations

- Risk: Route collisions or ambiguous slugs at root-level.<br>Mitigation: Fail CI/build on reserved-route collisions and validate slug/path mapping and precedence in tests.
- Risk: Invalid frontmatter breaking content rendering unexpectedly.<br>Mitigation: Enforce strict CI failure and clear local/dev warnings with source-file diagnostics.
- Risk: Unsafe MDX usage or unsupported components.<br>Mitigation: Restrict to an explicit approved MDX component map.
- Risk: Author confusion around required metadata.<br>Mitigation: Provide clear content examples and validation messages for required fields.
