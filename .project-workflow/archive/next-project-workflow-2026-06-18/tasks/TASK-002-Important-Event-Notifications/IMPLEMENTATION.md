## User Story

As the site owner, I want to receive email notifications when important customer events happen, so that I can respond quickly without manually checking multiple systems.

## Goal

- Deliver a safe internal event-notification workflow that emails the site owner about successful purchases, newsletter signups, payment-related failures, and success-link reissues, while also emitting Meta/Facebook signup events for successful newsletter signups.

## Approach

- Reuse the existing booking, enrollment, SES, and analytics integration points instead of introducing a separate event system.
- Add a small shared notification abstraction for internal event emails so event producers can send consistent payloads, multiple recipients can be supported centrally, and failures can be logged without breaking customer-facing flows.
- Implement the feature as vertical slices around event sources: booking and payment outcomes first, newsletter signup plus Meta tracking next, and success-link reissue plus cross-flow validation last.

## Phases

### Phase 1

- Changes: Establish the internal notification pipeline and recipient configuration, then wire it into successful purchase and payment-failure booking events with clear payload content and failure-safe behavior.
- Validation: Trigger successful purchase and payment-failure scenarios, verify internal emails reach all configured recipients with the expected identifiers, and confirm the customer-facing flow still succeeds if notification delivery fails.
- Tracker updates: Keep story status as `Analysing` while the plan is under review.

### Phase 2

- Changes: Extend notifications to newsletter signup and success-link reissue flows, add Meta/Facebook signup tracking for successful newsletter signups, and complete end-to-end validation for all supported event types.
- Validation: Submit successful signups from booking and content surfaces, trigger a success-link reissue, verify the internal emails and Meta signup event behavior, and confirm logging remains traceable across all supported events.
- Tracker updates: Move to `Plan Confirmed` only after explicit user approval.

## Files / Areas Likely to Change

- `app/api/booking/_lib/` for shared internal notification helpers, SES integration reuse, payment lifecycle hooks, and success-link reissue flows.
- `app/api/booking/payment-completion/route.ts` and `app/api/booking/stripe-webhook/route.ts` for successful purchase and failure event triggers.
- `app/api/booking/enrollment/route.ts` and `components/SendySignupForm.tsx` for newsletter signup notifications and Meta/Facebook tracking.
- Existing analytics surfaces such as `components/GoogleTagManager.tsx` or shared client-side tracking helpers if newsletter signup tracking should flow through GTM rather than direct pixel calls.

## Data / RLS / RPC / Migrations

- No database schema, RLS, RPC, or migration work is currently expected.
- Environment-backed configuration is likely needed for internal notification recipients and may need validation rules similar to existing email or analytics config.

## Risks & Mitigations

- Risk: Notification delivery failures could accidentally break customer-facing flows. Mitigation: keep notification sending non-blocking for supported user actions and log failures explicitly.
- Risk: Event hooks may fire more than once in booking lifecycle edge cases. Mitigation: attach notification triggers to the confirmed lifecycle points already treated as source-of-truth and validate duplicate behavior explicitly.
- Risk: Meta/Facebook signup tracking could diverge across booking and content surfaces. Mitigation: emit the signup tracking from the shared signup success path where possible.
- Risk: Multi-recipient configuration could be malformed or incomplete. Mitigation: validate recipient configuration early and fail in a controlled, observable way for the notification subsystem only.
